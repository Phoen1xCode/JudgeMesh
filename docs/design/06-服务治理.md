## 2. Nacos:注册中心 + 配置中心

### 2.1 注册中心

每个 Spring Boot 服务 `application.yaml`:

```yaml
spring:
  application:
    name: submit-service
  cloud:
    nacos:
      discovery:
        server-addr: nacos.judgemesh.svc:8848
        namespace: dev # dev / prod
        group: DEFAULT_GROUP
```

启动后 Nacos Console 可见服务列表,Gateway 通过 `lb://submit-service` 自动负载均衡。

### 2.2 配置中心

**集中管理**:数据库连接、JWT 密钥、限流阈值、特性开关。

```yaml
# Nacos 中的 submit-service.yaml
sentinel:
  flow:
    - resource: /api/submit
      grade: 1 # QPS
      count: 100
spring:
  datasource:
    url: jdbc:mysql://mysql.judgemesh.svc:3306/submits_db
    username: ${DB_USER}
    password: ${DB_PASS}
feature:
  contestFreezeMin: 30
  websocketEnabled: true
```

**热更新**:`@RefreshScope` 标记的 Bean 在配置变更时自动重加载,无需重启。

## 3. Sentinel:限流 / 熔断 / 降级

### 3.1 三层限流策略

| 层     | 维度     | 阈值       | 用途       |
| ------ | -------- | ---------- | ---------- |
| 网关层 | IP       | 10 QPS     | 防爬虫     |
| 网关层 | userId   | 30/min     | 防滥用提交 |
| 服务层 | 接口资源 | 各服务自定 | 保护下游   |

### 3.2 熔断规则示例

`submit-service` 调用 `user-service` 的 `getById`:

```java
@SentinelResource(
    value = "user-getById",
    fallback = "userFallback",   // 熔断时走 fallback
    blockHandler = "userBlocked" // 限流时走 blockHandler
)
public UserDTO getUser(Long id) {
    return userClient.getById(id);
}

public UserDTO userFallback(Long id, Throwable t) {
    return UserDTO.UNKNOWN; // 降级返回占位用户
}
```

熔断规则(Nacos 中):

```yaml
sentinel:
  degrade:
    - resource: user-getById
      grade: 0 # SLOW_REQUEST_RATIO
      count: 0.5 # 50% 慢调用
      timeWindow: 10 # 10s 后半开探测
      minRequestAmount: 5
      slowRatioThreshold: 0.5
```

### 3.3 答辩演示路径

老师问「你们怎么演示熔断?」:

1. 打开 Sentinel Dashboard
2. 用混沌注入让 user-service 30% 请求超时
3. submit-service 调用 user 的指标实时变红
4. 触发熔断后 fallback 返回占位数据,系统不挂
5. 10s 后探测恢复

## 4. Seata:分布式事务(AT 模式)

### 4.1 演示场景

**目的**:在系统里制造一个跨服务的强一致场景,演示 Seata。

**虚构需求**:用户每次提交扣 1 积分,AC 后退还 + 奖励。这不是真实 OJ 通用做法,但作为**分布式事务的教学样例**完美适配。

`submit-service` 中:

```java
@GlobalTransactional   // ← Seata 全局事务
public Long submit(SubmitRequest req) {
    // 分支 1:扣用户积分
    userClient.deductBalance(new DeductRequest(req.userId, 1));

    // 分支 2:写提交记录
    Submission s = submissionRepo.insert(buildSubmission(req));

    // 任意分支异常,Seata 自动回滚两边
    return s.getId();
}
```

### 4.2 Seata AT 模式工作原理(slides 用)

```
┌───────────┐  GlobalBegin   ┌────────────┐
│  TM(发起) │ ──────────────→│            │
│ submit-svc│                │  TC        │  ← Seata Server
│           │ ←──────────────│ (协调者)   │
└─────┬─────┘     XID        └─────┬──────┘
      │                            │
      │ 携带 XID                    │
      ↓                            │
┌───────────┐  RegisterBranch      │
│  RM       │ ────────────────────→│
│ user-svc  │                      │
│ (扣余额)   │ 业务+undo_log 同事务 │
│           │ ←────────────────────│
└─────┬─────┘                      │
      │                            │
      ↓                            │
┌───────────┐  RegisterBranch      │
│  RM       │ ────────────────────→│
│ submit-svc│                      │
│ (写记录)   │                      │
└─────┬─────┘                      │
      │                            │
      ↓ 异常?                      │
        ┌─────────────────────────┘
        ↓
    GlobalCommit / GlobalRollback
        │
        ↓
    所有 RM 提交本地事务 / 用 undo_log 反向回滚
```

### 4.3 部署形态

- Seata Server 部署在 K8s 集群内(单实例,生产应集群)
- 存储模式:用 MySQL 持久化(`global_table` / `branch_table` / `lock_table`)
- 注册到 Nacos,客户端从 Nacos 发现 TC 地址

## 5. JWT 鉴权

### 5.1 Token 设计

```json
{
  "sub": "12345",
  "username": "alice",
  "roles": ["STUDENT"],
  "iat": 1714400000,
  "exp": 1714403600
}
```

HS256 签名,密钥放 Nacos 配置中心(从环境变量注入)。
TTL:access token 1h,refresh token 7d。

### 5.2 验证流程

```
Browser ──[Authorization: Bearer xxx]──→ Gateway
                                          │
                                          ↓ 验签 + 检查过期
                                          ↓ 解析 sub / roles
                                          ↓
                                  注入 Header:
                                    X-User-Id: 12345
                                    X-User-Roles: STUDENT
                                          │
                                          ↓
                                  转发给下游服务
                                  (下游不重复验签,信任网关)
```

### 5.3 Refresh 机制

- access token 即将过期时(剩 < 5min),前端用 refresh token 换新 access token
- refresh token 单独存 Redis(`session:refresh:{userId}`),登出时主动失效

## 6. 混沌工程演示(ChaosMesh)

### 6.1 实验清单

| 实验                                  | 注入手段               | 期望表现                               |
| ------------------------------------- | ---------------------- | -------------------------------------- |
| **PodKill** 随机杀 Worker             | ChaosMesh PodChaos     | 任务 30s 内被重新派发                  |
| **PodKill** 杀 dispatcher leader      | ChaosMesh PodChaos     | etcd 10s 内触发新选主                  |
| **NetworkDelay** user-service 延迟 1s | ChaosMesh NetworkChaos | submit-service 触发慢调用熔断          |
| **CPUStress** Worker 节点 CPU 压满    | ChaosMesh StressChaos  | HPA 扩容,任务派发到其他节点            |
| **MQ 宕机** 杀 RabbitMQ Pod           | 手动 kubectl delete    | submit-service 走降级路径,前端友好提示 |

### 6.2 ChaosMesh 实验模板

```yaml
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: kill-judge-worker
  namespace: judgemesh
spec:
  action: pod-kill
  mode: one
  selector:
    namespaces: [judgemesh]
    labelSelectors:

```

## 7. 限流降级矩阵(全系统视角)

| 资源                      | 限流阈值   | 触发方              | 降级行为                 |
| ------------------------- | ---------- | ------------------- | ------------------------ |
| 全站 QPS                  | 1000       | Ingress             | 返回 429                 |
| 提交接口                  | 100 QPS    | Gateway Sentinel    | 排队 > 阈值返回 429      |
| 单用户提交                | 30/min     | Gateway Sentinel    | 返回提示                 |
| user-service.getById      | 200 QPS    | submit-svc Sentinel | fallback 占位用户        |
| problem-service.getDetail | 500 QPS    | gateway Sentinel    | 返回缓存结果             |
| dispatcher 派发           | 50 QPS     | dispatcher 内部     | MQ 排队                  |
| Worker /judge             | 10 QPS/Pod | Worker 内部         | 返回 503,dispatcher 重选 |

## 8. 健康探针

每个 K8s Deployment 配:

```yaml
livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 5
  periodSeconds: 5
```

Spring Boot Actuator 默认提供这两个探针,我们追加自定义检查:

- `submit-service`:MySQL + Redis + RabbitMQ 三个连接全部就绪才 Ready
- `judge-worker`:isolate 命令存在 + box 目录可写
