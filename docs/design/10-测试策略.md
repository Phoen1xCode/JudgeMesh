## 1. 测试金字塔

```
                       ┌─────────────┐

                       └──────┬──────┘
                              │

## 2. 单元测试

### 2.1 必测的核心逻辑

| 模块             | 测试点                              | 工具                 |
| ---------------- | ----------------------------------- | -------------------- |
| submit-service   | 状态机转换、计分公式、ZSet 写入逻辑 | JUnit 5 + Mockito    |
| problem-service  | 缓存 Cache-Aside、TTL 抖动          | JUnit 5              |
| user-service     | BCrypt、JWT 签发与验证              | JUnit 5              |
| judge-dispatcher | etcd 选主竞态、Worker 选择算法      | JUnit 5 + jetcd-test |
| judge-worker(Go) | isolate 输出解析、输出比对          | testing 包           |

### 2.2 覆盖率目标

| 模块                         | 行覆盖 | 分支覆盖 |
| ---------------------------- | ------ | -------- |
| 业务核心(状态机、计分、比对) | ≥ 85%  | ≥ 75%    |
| 普通业务代码                 | ≥ 60%  | ≥ 50%    |
| 框架胶水(Controller、Config) | 不强求 | —        |

CI 中用 JaCoCo 输出覆盖率报告。

## 3. 契约测试

服务间靠消息和 RPC 解耦,契约一旦改坏会导致下游崩。

### 3.1 测试点

| 契约                                        | 工具                              |
| ------------------------------------------- | --------------------------------- |
| submit-svc → MQ JudgeTask 消息              | 共享 schema 库 + JSON Schema 校验 |
| dispatcher → worker `/judge` 请求体         | OpenAPI 定义 + spectral lint      |
| worker → submit-svc `/internal/result` 回调 | OpenAPI                           |
| Feign UserClient.deductBalance              | Spring Cloud Contract(可选)       |

### 3.2 实施

把消息 / RPC 的 Java POJO 抽到 `judgemesh-api` 共享模块,每个服务依赖该模块。**任何改动都强制更新该模块版本号**,CI 检测下游服务是否还能编译。

## 4. 集成测试

    @Test
    void shouldAcceptSubmissionAndPublishToQueue() { ... }

    @Test
    void shouldUpdateStatusOnWorkerCallback() { ... }

    @Test
    void shouldRollbackWhenSeataAborts() { ... }   // 关键!
}
```

### 4.2 必测端到端用例

| 用例                                  | 期望                             |
| ------------------------------------- | -------------------------------- |
| 用户注册 → 登录 → 获取 token          | 200 + token 可用                 |
| 提交一道 AC 题 → 状态变 AC            | < 8s                             |
| 提交一道 WA 题 → 状态变 WA            | 输出对比正确                     |
| 提交一道 TLE 题                       | isolate 杀进程,状态 TLE          |
| 提交编译错误代码 → CE                 | judge_message 包含编译器输出     |
| 提交一道 SPJ 题(本期不做)             | (跳过)                           |
| Setter 上传题目 + 用例 → 学生提交可判 | 全链路                           |
| 比赛中提交 → 排行榜更新               | Redis ZSet 排名变化              |
| 封榜后提交 → 不推送排名变化           | WebSocket 静默                   |
| 重复提交(1 秒内同题)                  | 第二次 429                       |
| Seata 扣分异常时回滚                  | submission 不入库 + balance 不变 |

## 5. 安全测试(沙箱专项)

**故意写恶意代码,验证 isolate 拦得住**:

| 攻击                   | 代码                       | 期望                   |
| ---------------------- | -------------------------- | ---------------------- |
| Fork bomb              | C 写 while(1) fork()       | 超进程数限制,RE        |
| 写文件                 | `fopen("/etc/passwd","w")` | 权限拒绝,RE            |
| 死循环                 | C 写 `while(1);`           | TLE                    |
| 内存耗尽               | malloc 大数组              | MLE                    |
| 系统调用               | `execve("/bin/sh", ...)`   | seccomp 拦截,RE        |
| 联网                   | `socket(AF_INET, ...)`     | 拒绝,RE                |
| 子进程读取其他用户用例 | (在 box 外 ls)             | namespaces 隔离,见不到 |

这些用例放在 `tests/security/` 下,每次发版必跑。

## 7. 混沌测试

详见 [06-服务治理 §6](./06-服务治理.md)。

### 7.1 答辩前必演练 3 次的脚本

```bash
# 准备:启动后台压测
hey -z 5m -c 50 -m POST -H "Authorization: Bearer $TOKEN" \
    -D ./tests/fixtures/submit-payload.json \
    https://oj.judgemesh.example.com/api/submit &

# 实验 1:杀一个 Worker
kubectl delete pod -n judgemesh -l app=judge-worker --field-selector=status.phase=Running --limit=1
# 看大盘:1) Worker 数下降 2) 30s 内重新派发 3) HPA 在 2-3min 内拉新 Pod

# 实验 2:杀 dispatcher leader
LEADER=$(curl -s gateway/admin/dispatcher/status | jq -r '.leader')
kubectl delete pod -n judgemesh $LEADER
# 看大盘:1) is_leader 切换 2) 派发短暂停顿 3) 10s 内恢复

# 实验 3:RabbitMQ 故障
kubectl scale statefulset/rabbitmq -n judgemesh-infra --replicas=0
# 看大盘:1) submit-service 限流增多 2) Sentinel 触发降级
sleep 30
kubectl scale statefulset/rabbitmq -n judgemesh-infra --replicas=1
# 看大盘:恢复,堆积消息逐步消化
```

## 9. CI 流水线设计

```yaml
name: CI
on: [push, pull_request]

jobs:
  test:
    strategy:
      matrix:
        service: [user, problem, submit, dispatcher, worker, gateway]
    steps:
      - checkout
      - setup-jdk-17 / setup-go-1.22
      - run unit tests
      - run integration tests with testcontainers
      - upload coverage to codecov

  lint:
    - checkstyle / golangci-lint
    - openapi spectral lint

  build:
    needs: [test, lint]
    - docker buildx build & push

  deploy-dev:
    if: branch == main
    needs: build
    - kubectl apply -k overlays/dev
    - smoke test
```
