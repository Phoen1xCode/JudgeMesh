# 08 · 部署架构

## 1. 集群规划

### 1.1 节点角色

假设有 4 台云服务器(2C4G 起步,推荐 4C8G):

| 节点          | 角色              | 说明                               |
| ------------- | ----------------- | ---------------------------------- |
| node-master   | k8s control-plane | etcd / kube-apiserver / scheduler  |
| node-worker-1 | k8s worker        | 业务服务、可观测性                 |
| node-worker-2 | k8s worker        | 判题 worker(打 taint:`role=judge`) |
| node-worker-3 | k8s worker        | 判题 worker(同上)+ 中间件          |

> **学生预算紧张时**:可用 1 master + 2 worker 起步,后期再扩。
> 阿里云学生机/腾讯云学生机/华为云鲲鹏 99/年 一般够用。

### 1.2 网络

- CNI:Flannel(简单)或 Calico(支持 NetworkPolicy,推荐)
- Service 网段:10.96.0.0/12(默认)
- Pod 网段:10.244.0.0/16
- Ingress:`ingress-nginx` LoadBalancer 模式 → 公网入口走 master 节点 30080/30443 NodePort

### 1.3 命名空间

```
kube-system        K8s 自带
ingress-nginx      入口网关
judgemesh            ← 主业务
judgemesh-infra      ← MySQL/Redis/MQ/MinIO/etcd/Nacos/Seata
judgemesh-observe    ← Prometheus/Grafana/SkyWalking/Loki
chaos-mesh         混沌实验工具
```

## 2. K8s 资源清单(主要)

### 2.1 业务服务 Deployment 模板

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: submit-service
  namespace: judgemesh
spec:
  replicas: 2
  selector:
    matchLabels: { app: submit-service }
  template:
    metadata:
      labels: { app: submit-service }
      annotations:
        prometheus.io/scrape: "true"
        prometheus.io/path: /actuator/prometheus
        prometheus.io/port: "8083"
    spec:
      containers:
        - name: app
          image: registry/judgemesh/submit-service:${TAG}
          ports: [{ containerPort: 8083 }]
          env:
            - name: SPRING_PROFILES_ACTIVE
              value: prod
            - name: NACOS_ADDR
              value: nacos.judgemesh-infra.svc:8848
          resources:
            requests: { cpu: 200m, memory: 512Mi }
            limits: { cpu: 1000m, memory: 1Gi }
          livenessProbe:
            httpGet: { path: /actuator/health/liveness, port: 8083 }
            initialDelaySeconds: 30
          readinessProbe:
            httpGet: { path: /actuator/health/readiness, port: 8083 }
            initialDelaySeconds: 5
---
apiVersion: v1
kind: Service
metadata:
  name: submit-service
  namespace: judgemesh
spec:
  selector: { app: submit-service }
  ports:
    - port: 8083
      targetPort: 8083
```

### 2.2 judge-worker(带 HPA 与节点亲和)

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: judge-worker
  namespace: judgemesh
spec:
  replicas: 3
  selector: { matchLabels: { app: judge-worker } }
  template:
    metadata: { labels: { app: judge-worker } }
    spec:
      tolerations:
        - key: role
          operator: Equal
          value: judge
          effect: NoSchedule
      affinity:
        nodeAffinity:
          preferredDuringSchedulingIgnoredDuringExecution:
            - weight: 100
              preference:
                matchExpressions:
                  - { key: role, operator: In, values: [judge] }
      containers:
        - name: worker
          image: registry/judgemesh/judge-worker:${TAG}
          securityContext:
            privileged: true # isolate 需要
          resources:
            requests: { cpu: 1000m, memory: 1Gi }
            limits: { cpu: 2000m, memory: 2Gi }
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: judge-worker-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: judge-worker
  minReplicas: 3
  maxReplicas: 20
  metrics:
    - type: External
      external:
        metric:
          name: rabbitmq_queue_messages_ready
          selector: { matchLabels: { queue: submit.queue } }
        target:
          type: AverageValue
          averageValue: "5"
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300 # 缩容延迟 5min
    scaleUp:
      stabilizationWindowSeconds: 30
```

> **关于 `privileged: true`**:isolate 用 Linux namespaces + cgroups,需要较高权限。生产应该考虑改用更细粒度的 capabilities(SYS_ADMIN 等),本期先用特权简化。在 8.6 节标注为已知风险。

### 2.3 中间件 StatefulSet(略写)

`MySQL` / `Redis` / `MinIO` / `RabbitMQ` / `etcd` / `Nacos` 全部用 StatefulSet + PVC,本期单实例。

> **重要**:PVC 用云盘 StorageClass 持久化,**不要用 emptyDir**,否则 Pod 重启数据丢失。

### 2.4 Ingress

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: oj-ingress
  namespace: judgemesh
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "300"
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"
    cert-manager.io/cluster-issuer: letsencrypt-prod
spec:
  ingressClassName: nginx
  tls:
    - hosts: [oj.judgemesh.example.com]
      secretName: oj-tls
  rules:
    - host: oj.judgemesh.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend:
              service: { name: gateway, port: { number: 8080 } }
          - path: /ws
            pathType: Prefix
            backend:
              service: { name: gateway, port: { number: 8080 } }
          - path: /
            pathType: Prefix
            backend:
              service: { name: frontend, port: { number: 80 } }
```

> 演示时若没有真实域名,用 `oj.local` + hosts 文件即可,TLS 用自签证书。

## 3. CI/CD

### 3.1 流水线

```
Git push
  ↓
GitHub Actions
  ├─ 单元测试(每服务并行)
  ├─ 构建镜像(Buildx,multi-arch)
  ├─ 推送到镜像仓库
  ├─ 更新 K8s manifest 中的 image tag
  └─ kubectl apply -k overlays/prod/
```

### 3.2 镜像 tag 策略

- main 分支:`latest` + `git-sha`
- 发布:`v0.1.0`

### 3.3 配置仓库分层(Kustomize)

```
infra/k8s/
  base/
    user-service/
    problem-service/
    submit-service/
    judge-dispatcher/
    judge-worker/
    gateway/
  overlays/
    dev/      patches: 1 副本、本地镜像、minimal 资源
    prod/     patches: 多副本、HPA、资源配额
```

## 4. 配置与密钥

### 4.1 配置分层

| 层            | 内容                                      | 存储                           |
| ------------- | ----------------------------------------- | ------------------------------ |
| 镜像内        | 默认配置(profile=default)                 | 代码仓                         |
| Nacos         | 业务可调参数(限流阈值、特性开关)          | Nacos `${app}-${profile}.yaml` |
| K8s ConfigMap | K8s 相关配置(Nacos 地址、SkyWalking 地址) | 代码仓 + kustomize             |
| K8s Secret    | 数据库密码、JWT key、MinIO key            | sealed-secrets / 手动注入      |

### 4.2 密钥管理

**严禁**密钥进 Git。本期最简方案:

```bash
kubectl create secret generic db-credentials \
  --from-literal=username=oj_user \
  --from-literal=password=$(openssl rand -base64 24) \
  -n judgemesh
```

在 Deployment 中通过 `envFrom.secretRef` 注入。

> 进阶可考虑 Sealed Secrets 或 External Secrets,本期 stretch goal。

## 5. 可观测性栈部署

用社区 Helm chart,直接装即可:

```bash
# Prometheus + Grafana + Alertmanager
helm install prom prometheus-community/kube-prometheus-stack \
  -n judgemesh-observe --create-namespace \
  -f infra/helm/values/prom.yaml

# SkyWalking
helm install skywalking apache/skywalking \
  -n judgemesh-observe -f infra/helm/values/skywalking.yaml

# Loki
helm install loki grafana/loki-stack \
  -n judgemesh-observe -f infra/helm/values/loki.yaml
```

values 文件放在 `infra/helm/values/`,可版本控制。

## 6. ChaosMesh 部署

```bash
helm install chaos-mesh chaos-mesh/chaos-mesh \
  -n chaos-mesh --create-namespace \
  --set chaosDaemon.runtime=containerd
```

通过 CRD `PodChaos` / `NetworkChaos` 定义实验,放 `infra/chaos/`。

## 7. 启动顺序(首次部署)

```
1. K8s 集群就绪 + 安装 ingress-nginx + cert-manager
2. helm 装可观测性栈(observe namespace)
3. helm 装 ChaosMesh
4. apply 中间件:
   MySQL → Redis → MinIO → etcd → RabbitMQ → Nacos → Seata
   (顺序重要,Nacos 依赖 MySQL,Seata 依赖 Nacos)
5. 在 Nacos 中导入业务配置(`./scripts/nacos-import.sh`)
6. apply 业务服务:
   gateway → user → problem → submit → dispatcher → worker
7. apply Ingress
8. 健康检查脚本验证全链路:`./scripts/smoke-test.sh`
```

## 8. 已知风险与缓解

| 风险                              | 缓解                                                |
| --------------------------------- | --------------------------------------------------- |
| isolate 需要 privileged 容器      | 集群只用于本课作业,接受;答辩明确说明                |
| 中间件单实例,故障即数据风险       | 每日 cron 备份到 MinIO + 演示集群一旦炸了重新拉一次 |
| 学生云资源有限,HPA 扩到 20 跑不动 | demo 时把 maxReplicas 调到 8;答辩讲清楚理论上限     |
| 公网暴露需备案                    | 用内网 IP + 跳板机 / 仅在演示时开放                 |
| 镜像仓库带宽                      | 用阿里云个人镜像服务,VPC 内网拉取                   |

## 9. 灾备与回滚

- **代码回滚**:`kubectl rollout undo deployment/${name}`
- **配置回滚**:Nacos 自带版本,一键回滚
- **数据库迁移**:Flyway / Liquibase 管理,有 down 脚本
- **发布前**:每个服务新副本 readiness 通过后旧副本才被替换(K8s 默认 RollingUpdate)
