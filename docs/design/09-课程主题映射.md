## 第 1 讲:分布式软件概述

### 1.1 三高目标

- **高并发**(slides 原话「100 台机器支持 5 万并发」)
  - 落点:Worker 池 HPA 横向扩展;提交接口 P99 < 200ms
  - 大盘指标:`oj_submission_total{status="success"}` 每秒速率
- **高性能**(slides 原话「10 天任务 10 机器 1 天跑完」)
  - 落点:判题任务被 dispatcher 并行派发到多个 Worker
  - 大盘指标:`oj_judge_duration_seconds` 直方图
- **高可用**(slides 原话「10 台坏 1 台另外 9 台继续」)
  - 落点:Worker / dispatcher / 业务服务全部多副本;K8s 自愈
  - 演示:答辩混沌实验 UC-03

### 1.2 CAP / PACELC

参见 [02-总体架构 §3](./02-总体架构.md)。系统是 PA/EL 主流派,但分数据域差异化。**这是答辩区分度最高的回答之一** — 大多数学生只会说"我们是 AP 系统",而我们能精确到每个数据域。

### 1.3 BASE 理论

- **基本可用 BA**:任何单点故障都不会让"提交"功能完全不可用
- **软状态 S**:提交状态可短暂为 PENDING/JUDGING(中间态)
- **最终一致 E**:判题结果异步回写,最终所有副本数据一致

## 第 5 讲:分布式事务

### 3.1 演示场景:用户提交扣分(Seata AT)

参见 [06-服务治理 §4](./06-服务治理.md)。

### 3.2 答辩问答 Q&A 准备

> Q:为什么不用 2PC?
> A:2PC 同步阻塞,在网络分区时会无限等待协调者。Seata AT 本质上也是 2PC 的演化,但加了 undo_log 实现了"无侵入式"的反向补偿,且第二阶段是异步的。

> Q:为什么不用 TCC?
> A:TCC 需要业务层手写 Try / Confirm / Cancel,侵入性强。我们的扣分场景用 AT 更轻量。但若业务复杂(如对账、跨多步骤),TCC 更合适。

> Q:Seata AT 的核心原理是什么?
> A:每个 RM(资源管理器)在执行业务 SQL 前,先记录 before-image 到 undo_log,执行业务后记录 after-image。Commit 时异步删 undo_log;Rollback 时用 before-image 反向恢复。

## 第 6 讲:分布式共识

### 4.1 Raft 在项目中的体现

我们**用了 Raft,但没有自己实现**:

- etcd 内部用 Raft 共识保证 leader 选举的一致性
- 我们用 etcd 的 Lease + Compare-And-Swap 来实现 dispatcher 选主
- 这是工业界的标准做法,Kubernetes 自身就靠 etcd 选主

### 4.2 答辩可讲的 Raft 三阶段

> Q:讲一下 Raft 选主?
> A:
>
> 1. **Leader Election**:Follower 超时未收到心跳 → 转 Candidate → 投票 → 多数派同意成为 Leader
> 2. **Log Replication**:Leader 把日志发给所有 Follower,多数派 ack 后 commit
> 3. **Safety**:任期号 + 日志匹配性,保证已 commit 的日志不会丢失

> Q:为什么不自己实现?
> A:Raft 看似简单实则有 N 个边界条件(网络分区下的脑裂、日志冲突解决、快照、成员变更...),工业级实现至少几千行代码。我们的目标是**理解和应用**共识,不是重造一个 etcd。把时间花在业务亮点上,工程更明智。

### 4.3 答辩演示:杀 leader 看选主

```
1. kubectl exec -it dispatcher-0 -- curl localhost:8084/admin/dispatcher/status
   → 显示当前 leader 是 dispatcher-1
2. kubectl delete pod dispatcher-1
3. 观察 Grafana `oj_dispatcher_is_leader` 指标
   → dispatcher-2 在 ~10s 内变成新 leader
4. kubectl get pods → dispatcher-1 由 K8s 自动重建,加入 follower 行列
```

| 你的 Worker 是真的多机吗?                | 是,K8s 集群有 N 个 node,Worker Pod 通过节点亲和分布在 N 个 node 上。`kubectl get pods -o wide` 可证                                     |
| 如果 etcd 集群整体挂了怎么办?            | dispatcher 进入降级:保持当前 leader 身份继续工作,接受可能的"双主"风险(概率极低,因为 etcd 自身高可用)。这是 AP 取舍                      |

| 怎么测出 P99 8 秒这个数?                 | wrk + Lua 脚本压测,5 分钟 50 QPS,Prometheus 直方图按分位数统计。详见 [10-测试策略](./10-测试策略.md)                                    |
| 如果同时让你设计第二版,你会改什么?       | (1) 引入 ShardingSphere 分库;(2) Worker 按语言专池化避免编译器冲突;(3) 用 Knative 让 Worker scale-to-zero;(4) 加一个 Special Judge 服务 |
