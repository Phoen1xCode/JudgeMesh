## 5. 日志(Loki)

### 5.1 日志规范

**JSON 格式**(Logback 配置):

```json
{
  "timestamp": "2026-04-29T10:00:00.123Z",
  "level": "INFO",
  "service": "submit-service",
  "pod": "submit-service-7b9c-xyz",
  "traceId": "abc123",
  "spanId": "def456",
  "userId": 12345,
  "submissionId": 789,
  "message": "submission accepted",
  "extra": {...}
}
```

**关键字段**(必填):`service` / `traceId` / `userId`(若有)/ `submissionId`(若有)。

### 5.2 日志级别策略

| 级别  | 使用场景                       |
| ----- | ------------------------------ |
| ERROR | 系统异常,需立即关注            |
| WARN  | 业务异常但可恢复(如熔断、重试) |
| INFO  | 关键业务事件(登录、提交、AC)   |
| DEBUG | 仅开发环境开启                 |

**禁止**:

- 不打印用户密码、JWT、PII
- 单条日志体积 < 8KB,过大裁剪
- 不在循环里 INFO 级别打日志

### 5.3 跨服务日志关联

通过 `traceId` 串联:

- SkyWalking agent 自动注入 traceId 到 MDC
- Logback `%X{traceId}` 输出到 JSON
- Loki 查询 `{service="submit-service"} |~ "traceId=abc123"` 一键定位整条链路

## 6. 告警(AlertManager)

### 6.1 关键告警规则(Prometheus rules)

```yaml
groups:
  - name: oj-critical
    rules:
      - alert: SubmitServiceDown
        expr: up{job="submit-service"} == 0
        for: 1m
        annotations:
          summary: "submit-service 实例 {{ $labels.instance }} 不可达"

      - alert: DispatcherNoLeader
        expr: sum(oj_dispatcher_is_leader) < 1
        for: 30s
        annotations:
          summary: "判题调度器无主!提交将堆积"

      - alert: JudgeQueueBacklog
        expr: rabbitmq_queue_messages_ready{queue="submit.queue"} > 100
        for: 2m
        annotations:
          summary: "判题队列堆积超过 100,可能 Worker 不足"

      - alert: HighWAJudgeRate
        expr: rate(oj_judge_total{verdict="WA"}[5m]) / rate(oj_judge_total[5m]) > 0.95
        for: 5m
        annotations:
          summary: "WA 率异常高,可能用例文件错误"
```

## 7. 部署规模(K8s)

```yaml
prometheus: 1 replica, 50Gi PVC
grafana: 1 replica, ConfigMap 注入仪表盘
skywalking-oap: 2 replicas
skywalking-ui: 1 replica
loki: 1 replica, 20Gi PVC
promtail: DaemonSet (每节点 1 个)
alertmanager: 1 replica
```

资源占用:Observability stack 约占整个集群 3-4 GB 内存、2 vCPU。
