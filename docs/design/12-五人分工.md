## 设计原则

1. **@Phoen1xCode 完整承担前端**(28h 一次性投入)
2. **@Phoen1xCode 是团队 Go 主力**,所以 **judge-worker(Go)归 @Phoen1xCode**;@Phoen1xCode 也能写 Java,所以 API 网关与共享 API 模块也归 @Phoen1xCode
3. **后端工作量在 5 人之间尽量均分**:总后端 ~248h,人均 ~50h
4. **开发期高度独立**:每人对应一组清晰的代码目录,通过共享 API 模块约定接口
5. **难易搭配**:每人都有"主菜"(高复杂度 / 课程亮点)+"配菜"(基础 CRUD / 部署)
6. **关键路径冗余**:亮点模块至少有 1 名队友熟悉

## @Phoen1xCode · 组长 · 前端 + Go Worker + 网关

### 前端(主菜其一,~28h)

- React 18 + Vite + TypeScript 全套
- 状态管理:Zustand / Redux Toolkit;路由:React Router v6
- 页面:登录注册、题目列表 / 详情、提交框、提交记录、比赛列表、比赛页、排行榜、混沌演示页
- WebSocket 客户端封装(订阅单提交 + 比赛排行榜)
- UI 库:Ant Design / Material UI 任选

### judge-worker(Go,主菜其二,~22h)

团队 Go 主力,扛 worker:

- `/judge` 接口、任务并发管理(channel + worker pool)
- **isolate 沙箱**集成
- **4 种语言运行器**:C / C++ / Java / Python
- 用例下载(MinIO 预签名 URL)、输出严格比对
- 优雅停机(SIGTERM)、Prom 指标暴露(/metrics)
- Worker Docker 镜像(Ubuntu 22.04 + isolate + 4 语言工具链)

### Java 后端(配菜,~13h)

- **API 网关 Spring Cloud Gateway**(~10h):路由表、JWT 验证、Sentinel 限流接入、CORS、统一异常
- **共享 API 模块** `services/api/`(~5h):DTO / Feign 接口 / MQ schema 维护,**接口变更协调员**
- 与 @Nier291 协议联调(`/judge` 请求体 + `/internal/result` 回调)~3h(算入 worker 的测试时间)

> ⚠️ Worker 与 dispatcher 是关键耦合点。@Phoen1xCode 与 @Nier291 需要在 **Day 2 之前** 把 `JudgeTask` / `JudgeResult` 的 schema 定死,否则后续两边返工成本高。

## 协作规范

### Git

- main 受保护,所有改动走 PR + 至少 1 人 review
- 分支命名:`feat/${person}-${module}-${desc}`,如 `feat/a-worker-isolate-cpp`
- commit:`<type>(<module>): <desc>`,type ∈ {feat, fix, docs, refactor, test, chore}

### 沟通节奏

- **每周一次 1h 长会**:同步进度、对齐接口、分配下周
- **每天 15min 文字站会**:昨天做了 / 今天做 / blocker
- 重要决策沉淀到 commit message

### 接口约定流程

1. 提议方在 `services/api/` 模块改接口(@Phoen1xCode 守门)
2. 群里 @ 相关同学
3. 双方确认后合并到 main
4. 各自服务 pull 最新 API 版本

### Worker ↔ Dispatcher 协议(@Phoen1xCode↔@Nier291 重点关注)

@Phoen1xCode 和 @Nier291 **必须在第 9 周 Day 2** 把以下两个 schema 定死:

- dispatcher → worker:`POST /judge` 请求体
- worker → submit-service:`POST /api/submit/internal/result` 回调体

定下来后写到 `services/api/src/main/java/.../message/` 下,@Phoen1xCode 用 Go 这边按 JSON tag 实现对应 struct。任何一方改动必须 PR review,否则联调会反复返工。
