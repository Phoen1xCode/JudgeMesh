# 03 · 服务拆分与接口

## 1. 拆分原则

按 **领域驱动设计(DDD)聚合根** 划分:

- 每个服务管自己的核心数据,**不跨界写库**
- 跨服务读取走 OpenFeign(同步)或事件订阅(异步)
- 用户身份信息在 Gateway 解析 JWT,以 HTTP Header 透传(`X-User-Id`、`X-User-Roles`)

## 3. 各服务详细设计

### 3.1 gateway

**职责**:

1. 路由聚合:把路径前缀映射到下游服务
2. JWT 验证:解析 token,把 userId、roles 注入下游 Header
3. 限流:按 IP、按 userId 双维度限流(Sentinel)
4. CORS、统一异常包装、请求日志

**路由表**(Nacos 动态配置):

```yaml
routes:
  - id: user
    uri: lb://user-service
    predicates: [Path=/api/user/**, /api/auth/**]
  - id: problem
    uri: lb://problem-service
    predicates: [Path=/api/problem/**]
  - id: submit
    uri: lb://submit-service
    predicates: [Path=/api/submit/**, /api/contest/**, /api/rank/**]
  - id: ws
    uri: lb:ws://submit-service
    predicates: [Path=/ws/**]
```

**白名单**(免鉴权):

- `POST /api/auth/login`
- `POST /api/auth/register`
- `GET /api/problem/list`(允许游客浏览)

### 3.2 user-service

**职责**:用户身份管理 + 余额积分系统(用于演示分布式事务)

**核心实体**:

```java
class User {
    Long id;
    String username;
    String email;
    String passwordHash;  // BCrypt
    Integer balance;       // 用于 Seata 演示
    Set<Role> roles;
    LocalDateTime createdAt;
}

enum Role { GUEST, STUDENT, SETTER, ADMIN }
```

**对外 API**:

| 方法 | 路径                       | 说明                              |
| ---- | -------------------------- | --------------------------------- |
| POST | `/api/auth/register`       | 注册                              |
| POST | `/api/auth/login`          | 登录,返回 JWT                     |
| POST | `/api/auth/refresh`        | 刷新 token                        |
| GET  | `/api/user/me`             | 当前用户信息                      |
| GET  | `/api/user/{id}`           | 公开档案                          |
| POST | `/api/user/balance/deduct` | **Seata 演示**:扣分(参与全局事务) |

**对内 Feign 接口**(被 submit-service 调用):

```java
@FeignClient("user-service")
public interface UserClient {
    UserDTO getById(@PathVariable Long id);
    void deductBalance(@RequestBody DeductRequest req);  // 参与全局事务
}
```

### 3.3 problem-service

**职责**:题目元数据 + 测试用例存储

**核心实体**:

```java
class Problem {
    Long id;
    String title;
    String description;     // Markdown
    Integer timeLimit;       // ms
    Integer memoryLimit;     // MB
    Set<Tag> tags;
    Difficulty difficulty;
    Long setterId;
    Boolean published;
    String testcaseManifest;  // JSON: [{"input": "1.in", "output": "1.ans", "score": 10}, ...]
}

class TestCase {
    String inputObjectKey;   // MinIO key
    String outputObjectKey;
    Integer score;
}
```

**对外 API**:

| 方法 | 路径                                  | 说明                            |
| ---- | ------------------------------------- | ------------------------------- |
| GET  | `/api/problem/list`                   | 分页 + 搜索 + 标签筛选          |
| GET  | `/api/problem/{id}`                   | 题目详情                        |
| POST | `/api/problem`                        | 创建(Setter+)                   |
| PUT  | `/api/problem/{id}`                   | 编辑                            |
| POST | `/api/problem/{id}/testcase`          | 上传用例(multipart)             |
| GET  | `/api/problem/{id}/testcase/manifest` | **仅 Worker** 调用,获取用例清单 |

**缓存策略**:

- `problem:{id}` 题目详情 → Redis,TTL 1h,写入时失效
- `problem:list:hot` 热门题目列表 → Redis,TTL 5min,定时刷新

**MinIO 桶设计**:

- `testcases/problem-{id}/{caseIdx}.in`
- `testcases/problem-{id}/{caseIdx}.ans`

### 3.4 submit-service

**职责**:提交流程 + 比赛 + 排行榜 + WebSocket

**核心实体**:

```java
class Submission {
    Long id;
    Long userId;
    Long problemId;
    Long contestId;        // null = 平时提交
    Language language;     // C, CPP, JAVA, PYTHON
    String code;
    SubmissionStatus status;  // 状态机
    Integer score;
    Integer timeUsedMs;
    Integer memoryUsedKb;
    String judgeMessage;   // 编译错误信息或运行时错误
    Long judgedByWorker;   // 哪个 worker 判的(可观测性)
    LocalDateTime submittedAt;
    LocalDateTime judgedAt;
}

enum SubmissionStatus {
    PENDING, JUDGING, AC, WA, TLE, MLE, RE, CE, SE  // SE = System Error
}

class Contest {
    Long id;
    String title;
    LocalDateTime startTime;
    LocalDateTime endTime;
    Integer freezeBeforeMin;  // 封榜:结束前 N 分钟封榜
    List<Long> problemIds;
    Set<Long> registeredUserIds;
}
```

**对外 API**:

| 方法 | 路径                         | 说明                         |
| ---- | ---------------------------- | ---------------------------- |
| POST | `/api/submit`                | 提交代码                     |
| GET  | `/api/submit/{id}`           | 查询单个提交                 |
| GET  | `/api/submit/mine`           | 我的提交列表                 |
| GET  | `/api/contest/list`          | 比赛列表                     |
| POST | `/api/contest/{id}/register` | 报名                         |
| GET  | `/api/contest/{id}/rank`     | 比赛排行榜(从 Redis ZSet 读) |
| GET  | `/api/rank/global`           | 全站排行榜                   |
| WS   | `/ws/submission/{id}`        | 订阅单个提交状态             |
| WS   | `/ws/contest/{id}/rank`      | 订阅比赛排行榜更新           |

**Redis 数据结构**:

```
ZSET  rank:global                 → 成员=userId, 分数=AC题数*1e6 - 累计耗时(秒)
ZSET  rank:contest:{contestId}    → 成员=userId, 分数=ACM 计分公式
HASH  contest:{id}:freeze         → 封榜状态、封榜时刻
SET   contest:{id}:registered     → 已报名用户
PUB   ws:submission:{id}          → WebSocket 推送频道
PUB   ws:contest:{id}:rank        → 排行榜推送频道
```

**MQ 投递**(往 judge-dispatcher 发送):

```json
{
  "submissionId": 12345,
  "problemId": 100,
  "language": "CPP",
  "code": "...",
  "timeLimit": 1000,
  "memoryLimit": 256,
  "testcaseManifestUrl": "http://problem-service/api/problem/100/testcase/manifest",
  "submittedAt": "2026-04-29T10:00:00Z"
}
```

### 3.5 judge-dispatcher

**职责**:消费 MQ 任务 → 选 Worker → 派发 → 监控超时 → 重试

**多副本协调**:

- 启动时通过 etcd Lease 竞选 leader(key: `/judgemesh/dispatcher/leader`,TTL 10s)
- 仅 leader 消费 MQ 主队列;follower 监听 leader key,leader 失联时立即抢锁
- 所有副本共享派发日志(写 etcd `/judgemesh/dispatcher/inflight/{submissionId}`),便于 follower 接管

**Worker 选择算法**(简单加权随机):

1. 从 K8s API 拿到 `judge-worker` 标签下所有 Ready Pod 的 IP
2. 过滤掉过去 30s 内出现过 5xx 的节点(本地黑名单)
3. 从健康节点中按 (1 / 当前在跑任务数) 加权随机
4. 通过 HTTP / gRPC 推送任务到 Worker 的 `/judge` 端点

**超时与重试**:

- 派发后启动定时器,等待 Worker 主动回写
- 若 (timeLimit × 用例数 × 1.5 + 30s) 内未回写,认为 Worker 失联
- 任务回退到 MQ(带重试计数 +1),超过 3 次标记为 SE

**对外接口**:

```
GET /admin/dispatcher/status           当前 leader、各 follower 状态
POST /admin/dispatcher/chaos/kill-self  混沌测试用,主动放弃 leader
```

### 3.6 judge-worker(Go)

**职责**:接收任务 → isolate 沙箱 → 编译 → 跑用例 → 比对 → 回写

**Pod 内并发模型**:

- 每个 Pod 启动时根据 CPU 配额计算并发度(默认 = CPU \* 0.8)
- 用 channel 做内部任务队列,避免单个 Pod 被打爆
- 支持 graceful shutdown:收到 SIGTERM 后停止接新任务,等待在跑的完成或超时

**单任务执行流程**:

```
1. 接收 POST /judge,把请求写入本地任务通道
2. 从对象存储下载测试用例到 /tmp/judge-{submissionId}/
3. 把代码写到 /tmp/judge-{submissionId}/main.{ext}
4. 用 isolate --init 初始化一个独立的 box
5. 编译(对编译型语言):
   isolate --box-id=N --time=10 --mem=512MB --run -- gcc/javac/...
   失败 → 直接回报 CE
6. 对每个用例:
   isolate --box-id=N \
     --time=$timeLimit/1000 --wall-time=$timeLimit*2/1000 \
     --mem=$memoryLimit*1024 \
     --processes=1 \
     --no-default-dirs --dir=/box=./box \
     --stdin=in --stdout=out --stderr=err \
     --run -- ./a.out
   解析 isolate 元数据(返回码、CPU 耗时、内存峰值、退出原因)
   按规则比对输出 → AC / WA
7. 汇总所有用例结果,POST /api/submit/internal/result 回写
8. isolate --cleanup 清理 box
```

**对外接口**:

```
POST /judge        接收派发任务
GET  /healthz      就绪探针
GET  /metrics      Prometheus 指标
```

## 4. 服务间调用矩阵

| 调用方 → 被调方               | 同步方式                | 异步方式                | 数据        |
| ----------------------------- | ----------------------- | ----------------------- | ----------- |
| Browser → Gateway             | HTTPS                   | WS                      | 全部        |
| Gateway → user/problem/submit | LB                      | —                       | RESTful     |
| submit → user                 | Feign(查用户)           | —                       | UserDTO     |
| submit → problem              | Feign(查题目限制)       | —                       | ProblemDTO  |
| submit → judge-dispatcher     | —                       | RabbitMQ `submit.queue` | JudgeTask   |
| dispatcher → worker           | HTTP(派发)              | —                       | JudgeTask   |
| worker → submit               | HTTP `/internal/result` | —                       | JudgeResult |
| submit → user                 | Feign + Seata(扣分)     | —                       | (强一致)    |
| submit → Browser              | —                       | WebSocket               | 状态变更    |
