## 第 10 周:核心开发 + 文档(Sprint 1)

**目标**:每人主线功能跑通,需求 + 概要文档定稿。

### Day 1-3(周一~周三)各模块并行开发

| 模块                     | 第 10 周末必须完成                                                            | 负责  |
| ------------------------ | ----------------------------------------------------------------------------- | ----- |
| 前端                     | 登录 / 注册页、题目列表、提交框                                               | A     |
| 网关 + 共享 API          | JWT 验证、Sentinel 基础接入、Feign / MQ schema 定义                           | A     |
| **judge-worker**         | Go 服务骨架、`/judge` 接口、isolate 跑 C++ AC 用例                            | **A** |
| user-service             | 注册、登录、JWT 签发、`getById`、`deductBalance` 骨架                         | B     |
| problem-service          | 题目 CRUD、用例上传到 MinIO、Cache-Aside                                      | C     |
| submit-service           | `/submit` 接口、写库、投递 MQ、`/submit/{id}` 查询                            | D     |
| judge-dispatcher         | 消费 MQ、etcd 选主、调用 worker 接口(Mock)                                    | D     |

## 风险与缓解

| 风险                                | 概率 | 影响 | 缓解                                                                         |
| ----------------------------------- | ---- | ---- | ---------------------------------------------------------------------------- |
| K8s 学习曲线陡,第 9 周搭不起来      | 中   | 高   | 第 9 周 Day 2-3 全力搭,实在不行降级 Docker Compose,继续后续开发,第 12 周再试 |
| Seata 接入坑多                      | 中   | 中   | B 提前一周看官方 demo;实在不行降级为 TCC 手写或纯本地事务 + 文档说明         |
| isolate 在 K8s 里跑不通             | 中   | 高   | E 第 9 周末必须验证;不通则降级 Docker 容器 + cgroups                         |
| WebSocket 在 Ingress 后面有兼容问题 | 低   | 中   | Ingress 加 timeout 注解;实在不行用 SSE 退化                                  |
| 云服务器带宽不足                    | 低   | 中   | demo 时关闭非必要服务;镜像走 VPC 内网                                        |
| 5 人协作冲突                        | 中   | 中   | 周三周五例会;模块隔离边界清晰;急事拉群                                       |
| 压测时云资源费爆                    | 低   | 低   | 限定压测时长;注意关机                                                        |
| 比赛 / 期末 / 其他作业冲突          | 高   | 中   | 提前规划;每周固定 ≥ 12h 项目时间                                             |

## 总工时估算

| 角色   | 后端 | 前端 | 总       | 主要构成                                                                                      |
| ------ | ---- | ---- | -------- | --------------------------------------------------------------------------------------------- |

| B      | ~51h | —    | **~51h** | user 24 + Seata 客户端 6 + Seata Server 4 + DB 迁移 6 + MySQL 部署 4 + 指标接入 2 + 测试 5    |
| C      | ~46h | —    | **~46h** | problem 22 + MinIO 5 + 缓存 6 + 题目导入+demo 6 + 指标接入 2 + 测试 5                         |
| D      | ~57h | —    | **~57h** | submit 16 + 比赛 10 + WS+ZSet 12 + dispatcher 12 + 指标接入 2 + 测试 5                        |
| E      | ~54h | —    | **~54h** | CI/CD 8 + K8s 10 + Nacos 3 + 中间件部署 7 + 可观测 Helm 6 + 大盘 12 + 混沌 4 + HPA 2 + 联调 2 |

**后端总计 ~249h**,5 人均分约 50h。
**含前端总计 ~277h**,即每人每周 ~14h,**对大三课程负担可承受**。

> 后端工时差异:A=41 / B=51 / C=46 / D=57 / E=54,**标准差 ~6h(±12%)**。
> A 总工时最高(69h)是因为前端 28h 不可分摊;A 的后端虽然数字最低(41h),但 **judge-worker 是高复杂度核心模块**(沙箱安全 + 4 语言运行器),实际难度并不低。
> A 是团队 Go 主力,worker 必须 A 来扛;A 也写一部分 Java(网关 + 共享 API),与其他人协作衔接。
> D / E 后端略重是因为承担系统核心(提交链路 + 全部基础设施),合作时也最有"答辩讲点"。
