# 02 · 总体架构

> 对齐第 10 周「概要设计文档」总体章节

## 1. 设计原则

1. **单一职责**:每个微服务管自己的核心数据,不跨界写其他服务的库
2. **服务间松耦合**:同步调用走 OpenFeign + Sentinel;异步走 RabbitMQ
3. **基础设施无关**:服务本身不感知 K8s,通过 Spring Cloud Alibaba 抽象
4. **可观测优先**:每个服务从 Day 1 接入 Micrometer + SkyWalking
5. **YAGNI**:本期不做的功能(讨论区/题解等)就是不做,不留半成品坑

## 3. 一致性策略(CAP / PACELC 落地)

回顾课程第 1 讲:

> **PACELC**:分区(P)时在 @Phoen1xCode/@KY-raika 之间权衡;否则(@sigufh)在 L/@KY-raika 之间权衡。

本系统**不是统一的一种策略**,而是按数据域差异化:

| 数据域                    | 一致性级别           | PACELC | 理由                            |
| ------------------------- | -------------------- | ------ | ------------------------------- |
| 用户账号 / 密码           | 强一致(MySQL ACID)   | PC/EC  | 安全相关,允许慢一点             |
| 用户余额 / 积分(操作链路) | 强一致(Seata AT)     | PC/EC  | 演示分布式事务,扣分必须不丢不重 |
| 题目元数据                | 强一致 + 缓存        | PC/EL  | 写少读多,写穿 + 失效缓存        |
| 测试用例文件              | 强一致(MinIO)        | PC/EC  | 二进制不允许出错                |
| 提交记录                  | 强一致(MySQL)        | PC/EC  | 用户的「成绩单」,不能丢         |
| 判题结果回写              | 最终一致(MQ + 重试)  | PA/EL  | 允许短暂 Pending,最终必到       |
| 排行榜                    | 最终一致(Redis ZSet) | PA/EL  | 用户对短暂排名滞后宽容          |
| WebSocket 推送            | 至多一次             | PA/EL  | 推送失败客户端会主动拉取        |

## 5. 关键技术决策(ADR 摘要)

### ADR-001:为什么选 Spring Cloud Alibaba 而非 Netflix OSS?

**背景**:课程 slides 第 1 讲 / 第 7 讲提到 Eureka、Hystrix、Ribbon 等 Netflix 组件。
**决策**:采用 Spring Cloud Alibaba(Nacos / Sentinel / Seata)。
**理由**:

- Netflix Eureka 自 2018 年起进入维护模式,不再有重大更新
- Hystrix 已停止开发,官方推荐迁移到 Resilience4j 或 Sentinel
- Spring Cloud Netflix 项目已从 Spring Cloud 主线移除
- Alibaba 全家桶在国内中文文档最全,Seata 还能补上「分布式事务」这一讲
- 答辩时可表述为「我们选择了课件中介绍的 Spring Cloud 治理能力的现代云原生演进版本」

### ADR-002:为什么 dispatcher 用 etcd 选主而非自实现 Raft?

**决策**:用 etcd Lease + Watch 做主备选举。
**理由**:

- 自实现 Raft 至少需要 1 人 1 周的纯算法工作量,边际收益低
- etcd 本身就是 Raft 的工业级实现,「站在巨人肩膀上」是现代分布式系统的正确姿态
- 课件第 6 讲讲的也是 Raft 的概念,不要求自实现
- 答辩可清晰表述:「我们用 etcd 这个经过 Kubernetes 验证的 Raft 实现,把共识能力作为基础设施引入」

### ADR-003:为什么判题用 K8s Worker Pod 池而非 K8s Job per submission?

**决策**:固定 Worker Pod 池 + Pod 内并发多个 isolate 实例。
**理由**:

- K8s Job 启动需要 1-3 秒,远高于 isolate 启动的 30-100ms
- 高并发场景下,频繁创建 Pod 会让 K8s API Server 不堪重负
- 固定 Pod 池配合 HPA 弹性伸缩,既覆盖「容器编排」又覆盖「弹性伸缩」主题
- isolate 已经提供操作系统级的隔离,不需要 Pod 级隔离

### ADR-004:为什么 Worker 用 Go 而其他服务用 Java?

**决策**:judge-worker 单独使用 Go,其他服务统一 Java。
**理由**:

- Worker 是计算密集 + 大量子进程操作,Go 的 goroutine 比 Java 线程模型更轻
- 调用 isolate CLI 时,Go 起子进程的开销是 Java 的 1/10
- 团队组长 @Phoen1xCode 是 Go 主力,自然由 @Phoen1xCode 写 worker;@Phoen1xCode 也能写 Java,所以负责网关与共享 API 模块
- Worker 与其他服务的耦合面是 MQ 消息 + HTTP 回调,语言异构成本可控
- 「按场景选语言」是答辩亮点,体现工程判断力
