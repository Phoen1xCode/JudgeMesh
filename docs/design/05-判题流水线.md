## 2. 状态机

```
                ┌─────────────────────────────────┐
                │                                 │
                │             [SE 重试 3 次后]      │
                ↓                                 │
              PENDING ──→ JUDGING ──→ AC          │
                ↑              │   ──→ WA          │
                │              │   ──→ TLE         │
                │ 重试         │   ──→ MLE         │
                │              │   ──→ RE          │
                └──────────────│   ──→ CE          │
                               │   ──→ SE ─────────┘
                               (终态:AC/WA/TLE/MLE/RE/CE)
```

| 状态    | 含义                                  | 终态? |
| ------- | ------------------------------------- | ----- |
| PENDING | 已提交,等待派发                       | ✗     |
| JUDGING | Worker 已收,正在跑                    | ✗     |
| AC      | Accepted,通过所有用例                 | ✓     |
| WA      | Wrong Answer                          | ✓     |
| TLE     | Time Limit Exceeded                   | ✓     |
| MLE     | Memory Limit Exceeded                 | ✓     |
| RE      | Runtime Error                         | ✓     |
| CE      | Compile Error                         | ✓     |
| SE      | System Error,系统异常(用例文件丢失等) | ✓     |

## 3. 关键设计点

### 3.1 dispatcher 多副本协调(Raft via etcd)

**为什么要多副本**:

- 单 dispatcher 是单点故障,挂了所有提交都卡在 PENDING
- 但所有副本同时消费 MQ 会导致重复派发

**解决方案** — etcd Lease 选主 + 仅 leader 消费:

```
启动时:
  for podName in pod_list:
    1. lease = etcd.grantLease(ttl=10s)
    2. 尝试 etcd.Txn:
         If(/judgemesh/dispatcher/leader 不存在)
            Then put(/judgemesh/dispatcher/leader, podName, lease)
         成功 → 我是 leader
         失败 → 我是 follower
    3. follower 注册 /judgemesh/dispatcher/members/{podName}
       并 watch leader key

leader 工作循环:
  while True:
    1. 续约 lease(每 3s 一次)
    2. 消费 MQ 一条任务
    3. 选 worker(下文 §3.2)
    4. etcd.Put(/judgemesh/dispatcher/inflight/{submissionId}, {worker, ts})
    5. POST 到 worker /judge

leader 失联(进程崩、网络分区):
  → lease 10s 不续就过期
  → leader key 自动删除
  → 所有 follower 收到 watch 事件,立即抢锁
  → 新 leader 扫描 inflight,接管超时任务
```

**异常处理**:

- 旧 leader 假死复活后会发现自己 lease 已过期,放弃职责重新竞选
- inflight key 也带 lease,不会留下脏数据

### 3.2 Worker 选择算法

```python
def select_worker(task):
    # 1. 拿到所有就绪 Pod
    pods = k8s_api.list_pods(
        namespace="judgemesh",
        label_selector="app=judge-worker",
        field_selector="status.phase=Running"
    )
    candidates = [p for p in pods if is_ready(p)]

    # 2. 过滤本地黑名单(过去 30s 5xx)
    candidates = [p for p in candidates if p.ip not in blacklist]

    if not candidates:
        raise NoAvailableWorker()  # 触发重试

    # 3. 加权随机:权重 = 1 / (当前在跑任务数 + 1)
    weights = [1.0 / (inflight_count.get(p.ip, 0) + 1) for p in candidates]
    return weighted_random_choice(candidates, weights)
```

### 3.3 Worker 内执行细节(isolate)

**编译阶段**(以 C++ 为例):

```bash
isolate --box-id=$N --init
cp main.cpp /var/local/lib/isolate/$N/box/
isolate --box-id=$N \
        --time=10 --wall-time=15 \
        --mem=524288 \
        --processes=10 \
        --stderr-to-stdout \
        --run -- /usr/bin/g++ -O2 -std=c++17 main.cpp -o main 2>&1
```

**运行阶段**(每个用例):

```bash
isolate --box-id=$N \
        --time=$((tl_ms / 1000)).$((tl_ms % 1000)) \
        --wall-time=$((tl_ms * 2 / 1000)) \
        --mem=$((ml_mb * 1024)) \
        --processes=1 \
        --no-default-dirs \
        --dir=box=./box \
        --stdin=in.txt \
        --stdout=out.txt \
        --stderr=err.txt \
        --meta=meta.txt \
        --run -- ./main
```

**isolate meta 输出解析**:

```
time:0.234              # CPU 秒
time-wall:0.512         # 墙钟秒
max-rss:25600           # KB
status:OK               # OK / TO(超时) / SG(信号) / RE(非零退出) / XX(系统错)
exitcode:0
```

按 status 映射:
| isolate status | 我们的状态 |
|---|---|
| OK + 输出匹配 | AC |
| OK + 输出不匹配 | WA |
| TO | TLE |
| SG(SIGKILL,通常因 OOM) | MLE |
| RE | RE |
| XX | SE |

### 3.4 输出比对策略

**默认严格比对**(本期不实现 SPJ):

```python
def compare(out: bytes, ans: bytes) -> bool:
    # 1. 去除 windows \r\n → \n
    out = out.replace(b'\r\n', b'\n')
    ans = ans.replace(b'\r\n', b'\n')
    # 2. 去除末尾空白(很多人输出末尾多个回车)
    out = out.rstrip()
    ans = ans.rstrip()
    return out == ans
```

## 4. 性能与容量

### 4.1 Pod 内并发设计

每个 Worker Pod 资源配额(K8s):

```yaml
resources:
  requests:
    cpu: 1000m
    memory: 1Gi
  limits:
    cpu: 2000m
    memory: 2Gi
```

并发度 = `floor(cpu.limit * 0.8)` = 1.6 ≈ **1 个并发 isolate**

为什么要这么保守?

- isolate 自身要占一些 CPU
- 同 Pod 内多 isolate 会互相争抢内存带宽,影响判题精度
- 把扩展性交给 Pod 层(HPA 加 Pod),而不是 Pod 内并发

### 4.2 HPA 弹性伸缩

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler

### 4.3 容量估算

- 平时:平均 5 提交/秒,Worker 利用率 ~30%
- 比赛:峰值 50 提交/秒(瞬时),队列削峰后 Worker 持续吃 ~20/秒
- 单 Worker 处理能力:~2 个提交/秒(每个 ~500ms,简单题)
- 比赛峰值期需 ~10 个 Worker → HPA 在 minReplicas=3 → maxReplicas=20 之间游动

## 5. 故障与降级

| 故障场景               | 表现                          | 处理                                                                         |
| ---------------------- | ----------------------------- | ---------------------------------------------------------------------------- |
| 单 Worker 宕机         | 该 Worker 上的任务 30s 未回写 | dispatcher 重新派发                                                          |
| 所有 Worker 宕机       | 队列堆积 > 阈值               | Sentinel 在网关层限流提交接口;前端友好提示                                   |
| dispatcher leader 宕机 | etcd lease 10s 后过期         | follower 接管                                                                |
| etcd 不可用            | 选主失效                      | 降级:**保留当前 leader 继续工作**(可能短暂双主,接受概率不高);etcd 恢复后重选 |
| MQ 不可用              | 提交无法异步化                | submit-service 直接同步调 dispatcher 的紧急端点(降级,不限流)                 |
| MySQL 不可用           | 提交无法持久化                | 网关层 Sentinel 熔断,返回 503;监控告警                                       |
| MinIO 不可用           | 拉不到用例                    | Worker 标记 SE,重试 3 次后失败;Setter 上传也会失败                           |

## 6. 防作弊与限流(MVP 范围)

- **同用户同题 1 秒内不能重复提交**:Redis `SET dedup:submit:{userId}:{problemId} 1 NX EX 1`
- **同用户每分钟最多 30 次提交**:Sentinel 在网关层 `userId` 维度
- **代码长度限制**:64 KB
- **比赛中只能提交本场比赛题目**
