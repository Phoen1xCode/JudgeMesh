# 04 · 数据模型与存储

## 1. 数据归属总览

| 存储                | 实例       | 主要数据                    | 读写特征           |
| ------------------- | ---------- | --------------------------- | ------------------ |
| MySQL `users_db`    | 单实例     | 用户、角色、余额            | 写少读多           |
| MySQL `problems_db` | 单实例     | 题目元数据                  | 写极少读极多       |
| MySQL `submits_db`  | 单实例     | 提交记录、比赛              | 写多(峰值高)+ 读多 |
| Redis(单实例 7.x)   | 单实例     | 排行榜、缓存、会话、Pub/Sub | 极高频读写         |
| MinIO(单节点)       | 单实例     | 测试用例文件                | 一次上传,多次只读  |
| etcd(3 节点)        | 3 节点共识 | dispatcher 选主、分布式锁   | 极少               |
| RabbitMQ(单实例)    | 单实例     | 提交任务 / 重试 / DLQ       | 高频写             |

> **本期不实现 MySQL 主从、Redis Cluster、Kafka 等**,在 [02-总体架构 §6](./02-总体架构.md) 已说明。

## 2. MySQL Schema

### 2.1 users_db

```sql
CREATE TABLE `user` (
    `id`            BIGINT      NOT NULL AUTO_INCREMENT,
    `username`      VARCHAR(64) NOT NULL UNIQUE,
    `email`         VARCHAR(128) NOT NULL UNIQUE,
    `password_hash` CHAR(60)    NOT NULL,         -- BCrypt
    `nickname`      VARCHAR(64),
    `avatar_url`    VARCHAR(255),
    `balance`       INT         NOT NULL DEFAULT 100,  -- 用于 Seata 演示
    `total_ac`      INT         NOT NULL DEFAULT 0,
    `total_submit` INT          NOT NULL DEFAULT 0,
    `created_at`    DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at`    DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    `deleted_at`    DATETIME,                     -- 软删
    PRIMARY KEY (`id`),
    INDEX `idx_email` (`email`)
);

CREATE TABLE `user_role` (
    `user_id` BIGINT     NOT NULL,
    `role`    VARCHAR(16) NOT NULL,  -- STUDENT / SETTER / ADMIN
    PRIMARY KEY (`user_id`, `role`)
);

-- Seata 必需的 undo_log 表(AT 模式自动管理)
CREATE TABLE `undo_log` (
    `branch_id`     BIGINT    NOT NULL,
    `xid`           VARCHAR(128) NOT NULL,
    `context`       VARCHAR(128) NOT NULL,
    `rollback_info` LONGBLOB  NOT NULL,
    `log_status`    INT       NOT NULL,
    `log_created`   DATETIME  NOT NULL,
    `log_modified`  DATETIME  NOT NULL,
    UNIQUE KEY `ux_undo_log` (`xid`, `branch_id`)
);
```

### 2.3 submits_db

```sql
CREATE TABLE `submission` (
    `id`              BIGINT      NOT NULL AUTO_INCREMENT,
    `user_id`         BIGINT      NOT NULL,
    `problem_id`      BIGINT      NOT NULL,
    `contest_id`      BIGINT,                     -- null = 平时提交
    `language`        VARCHAR(16) NOT NULL,
    `code`            MEDIUMTEXT  NOT NULL,
    `code_length`     INT         NOT NULL,
    `status`          VARCHAR(16) NOT NULL DEFAULT 'PENDING',
    `score`           INT         NOT NULL DEFAULT 0,
    `time_used_ms`    INT,
    `memory_used_kb`  INT,
    `judge_message`   TEXT,                        -- 编译错误或运行时输出
    `judged_by_worker` VARCHAR(64),                -- Worker pod 名,可观测性
    `submitted_at`    DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `judged_at`       DATETIME,
    PRIMARY KEY (`id`),
    INDEX `idx_user_submitted` (`user_id`, `submitted_at`),
    INDEX `idx_problem_status` (`problem_id`, `status`),
    INDEX `idx_contest_user` (`contest_id`, `user_id`)
);

CREATE TABLE `contest` (
    `id`           BIGINT      NOT NULL AUTO_INCREMENT,
    `title`        VARCHAR(255) NOT NULL,
    `description`  TEXT,
    `start_time`   DATETIME    NOT NULL,
    `end_time`     DATETIME    NOT NULL,
    `freeze_min`   INT         NOT NULL DEFAULT 30,
    `created_by`   BIGINT      NOT NULL,
    `created_at`   DATETIME    NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`id`),
    INDEX `idx_time` (`start_time`, `end_time`)
);

CREATE TABLE `contest_problem` (
    `contest_id` BIGINT NOT NULL,
    `problem_id` BIGINT NOT NULL,
    `seq`        INT    NOT NULL,                 -- A B C D ... 的次序
    PRIMARY KEY (`contest_id`, `problem_id`)
);

CREATE TABLE `contest_register` (
    `contest_id`    BIGINT   NOT NULL,
    `user_id`       BIGINT   NOT NULL,
    `registered_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (`contest_id`, `user_id`)
);
```

## 3. Redis 数据结构

| Key 模式                    | 类型         | 用途                                        | TTL              |
| --------------------------- | ------------ | ------------------------------------------- | ---------------- |
| `cache:problem:{id}`        | STRING(JSON) | 题目详情缓存                                | 1h,写穿失效      |
| `cache:problem:list:hot`    | STRING(JSON) | 热门题目首页缓存                            | 5min             |
| `cache:user:{id}`           | STRING(JSON) | 用户公开档案缓存                            | 30min            |
| `rank:global`               | ZSET         | 全站排行榜                                  | 持久             |
| `rank:contest:{cid}`        | ZSET         | 比赛排行榜                                  | 比赛结束后 +7 天 |
| `contest:{cid}:freeze`      | HASH         | 封榜状态 `frozen=true/false` `frozenAt=...` | 比赛结束后 +1 天 |
| `contest:{cid}:registered`  | SET          | 报名用户集合                                | 比赛结束后 +1 天 |
| `lock:problem:{id}`         | STRING + NX  | 题目编辑分布式锁                            | 30s              |
| `dedup:submit:{userId}`     | STRING + NX  | 防重提交,1 秒 1 次                          | 1s               |
| `pubsub:submission:{id}`    | PUB/SUB      | 单提交状态变更频道                          | —                |
| `pubsub:contest:{cid}:rank` | PUB/SUB      | 比赛排行榜更新频道                          | —                |

### 3.1 排行榜 ZSet 计分公式

**全站榜**:

```
score = (totalAC * 1_000_000) - totalAcceptedTimeSeconds
```

AC 多的排前面;AC 数相同时,总耗时短的排前面。

**比赛榜(ACM 风格)**:

```
对每个用户:
  solved      = 已 AC 题数
  penalty_min = sum(每道 AC 题:首次 AC 时间(分钟) + 错误提交数 * 20)

score = (solved * 100_000) - penalty_min
```

保证 ZSet 中分数大者排前。

### 3.2 排行榜更新策略

- **写入路径**:judge-worker 回写 `AC` → submit-service 触发更新
  ```java
  if (newStatus == AC && !alreadyAC(user, problem)) {
      redisTemplate.opsForZSet().incrementScore("rank:global", userId, 1_000_000 - elapsedSeconds);
      redisTemplate.convertAndSend("pubsub:contest:" + cid + ":rank", delta);
  }
  ```
- **读取路径**:WebSocket 客户端订阅频道 + 首次连接时拉一次全量
- **封榜**:WebSocket 服务端在封榜后过滤推送(只推送私榜 / 不推送)

## 4. MinIO 桶结构

```
testcases/
  problem-{id}/
    1.in
    1.ans
    2.in
    2.ans
    ...
    manifest.json    -- 清单(冗余,加速 Worker 拉取)

avatars/
  user-{id}.png

backups/
  daily/
    2026-04-29/
      users.sql.gz
      problems.sql.gz
      submits.sql.gz
```

**访问控制**:

- `testcases/` 桶设为私有,Worker 用预签名 URL 拉取(预签名有效期 5min)
- `avatars/` 桶设为公开读

## 5. RabbitMQ 拓扑

```
Exchange:  judgemesh.exchange  (type=direct, durable)

Queues:
  submit.queue           ← submit-service 投递,dispatcher 主节点消费
    routing-key: judge.task
    DLX → submit.dlq

  submit.retry.queue     ← dispatcher 重试投递(带延迟)
    routing-key: judge.retry
    x-message-ttl: 5000  (5s 后回到主队列)
    DLX → submit.queue

  submit.dlq             ← 最终失败队列
    人工介入 / 报警

  result.queue           ← worker 回报结果(可选,目前用 HTTP 回调)
    routing-key: judge.result
```

**消息格式**:见 [03-服务拆分 §3.4](./03-服务拆分.md) 末尾 MQ 投递示例。

## 6. etcd Key 设计

```
/judgemesh/
  dispatcher/
    leader              -- value=podName, lease=10s, 选主用
    members/
      {podName}         -- 心跳,follower 注册

    feature-flags       -- 灰度开关(可选)
```

## 7. 数据生命周期与备份

| 数据类          | 保留时间     | 备份策略                                                 |
| --------------- | ------------ | -------------------------------------------------------- |
| 用户、题目      | 永久         | 每日 mysqldump → MinIO `backups/daily/`                  |
| 提交记录        | 永久         | 同上                                                     |
| 测试用例        | 永久         | MinIO 自身版本 + 定期同步                                |
| Redis 排行榜    | 持久         | 每天定时从 MySQL 重建快照(防止 Redis 数据丢失后排名错乱) |
| RabbitMQ 队列   | 仅未消费消息 | 持久化到磁盘(`durable: true` + 消息 `persistent`)        |
| 日志(Loki)      | 7 天         | 自动滚动,过期清理                                        |
| Prometheus 指标 | 15 天        | 自动滚动                                                 |
