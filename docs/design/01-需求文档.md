# 01 · 需求文档

> 对齐第 10 周「需求文档」检查点

## 1. 项目背景

在线判题(Online Judge, OJ)系统是 ACM/ICPC、CCPC、蓝桥杯、各高校算法课程的核心基础设施。其典型负载特征——**高并发的代码提交、计算密集的判题任务、流量陡然倾斜的比赛场景**——使其成为「分布式三高」的天然示范场景:

- **高并发**:比赛开始 1 分钟内可能涌入数千次提交,瞬时 QPS 远高于平时
- **高性能**:判题需要高效调度到空闲计算节点,平均判题延迟需在 1-2 秒
- **高可用**:任何一个判题节点宕机不应影响整个系统;比赛期间任何停服都是事故

本项目以 OJ 系统为载体,落地课程中讲授的微服务架构、服务治理、分布式共识、CAP/BASE、可观测性、容器编排等核心知识点。

## 4. 非功能性需求

### 4.1 性能指标(可压测验证)

| 指标                | 目标                              | 测量方法     |
| ------------------- | --------------------------------- | ------------ |
| 提交接口 P99 延迟   | < 200ms(同步入队部分)             | wrk / JMeter |
| 判题端到端 P50 延迟 | < 2 秒(C++ 简单题)                | 业务埋点     |
| 判题端到端 P99 延迟 | < 8 秒                            | 业务埋点     |
| 系统提交吞吐        | > 100 QPS(常态) / > 500 QPS(比赛) | 压测         |
| 排行榜查询 P99      | < 50ms                            | wrk          |

### 4.2 可用性指标

| 指标                 | 目标                                            |
| -------------------- | ----------------------------------------------- |
| 任意单 Worker 宕机   | 不影响系统可用,30 秒内任务被重新派发            |
| 任意单微服务实例宕机 | 网关自动剔除,5 秒内流量切到其他副本             |
| 调度器主节点宕机     | 10 秒内 etcd 选出新主,继续派发                  |
| MySQL/Redis 维护     | 暂不要求高可用(单实例 + 备份),作为 stretch goal |

### 4.3 一致性策略(本系统的 PACELC 定位)

本系统整体定位为 **PA / EL**(分区时优先可用,正常时优先低延迟):

- 提交记录 → MySQL(强一致核心)
- 排行榜 → Redis ZSet(弱一致 + 最终一致)
- 判题结果 → 异步回写 + 事件驱动(BASE)
- 用户余额扣分类操作 → Seata AT 模式(强一致演示)

详见 [02-总体架构](./02-总体架构.md) §3。

### 4.4 安全需求

| 维度     | 要求                                                                                   |
| -------- | -------------------------------------------------------------------------------------- |
| 鉴权     | JWT(网关层验证)+ RBAC                                                                  |
| 沙箱     | isolate 提供 cgroups + namespaces + seccomp 隔离;CPU、内存、进程数、磁盘、网络全部限制 |
| SQL 注入 | MyBatis-Plus 参数化                                                                    |
| XSS      | 前端转义 + CSP 头                                                                      |
| 限流     | Sentinel 在网关层按 IP / userId 限流                                                   |

## 5. 关键用例

### 5.1 用例 UC-01:学生提交代码

**主流程**:

1. 学生在题目页选择语言、写代码、点击「提交」
2. 前端 → Gateway → submit-service
3. submit-service 写 `submission` 表(状态=Pending),投递 MQ
4. 返回提交 ID 给前端
5. 前端用 WebSocket 订阅该提交 ID
6. judge-dispatcher(主节点)消费 MQ,通过 K8s API / 内置算法选 Worker
7. judge-worker 拉取代码 + 用例,在 isolate 沙箱内编译运行
8. Worker 比对输出,回写结果到 submit-service
9. submit-service 更新 DB 状态,通过 WebSocket 推送给前端

**异常分支**:

- Worker 超时未回报 → dispatcher 30s 后重新派发,最多 3 次
- 编译失败 → 状态 = CE,记录编译器输出
- 沙箱告警(尝试越权)→ 状态 = RE + 安全标记

### 5.2 用例 UC-02:比赛实时排行榜

**主流程**:

1. 用户加入比赛后,前端订阅比赛排行榜 WebSocket 频道
2. 任何人 AC 一题,judge 回写时同步更新 Redis ZSet `contest:{id}:rank`
3. submit-service 监听 Redis 变更(或主动推送),向 WebSocket 频道广播 delta
4. 所有订阅该频道的客户端实时刷新

**封榜机制**:

- 比赛剩余 < 30min 时,服务端不再向客户端推送排名变化
- 提交本身仍正常处理,只是榜单冻结显示
- 比赛结束后,管理员手动「解榜」

### 5.3 用例 UC-03:答辩现场混沌演示

**演示流程**(老师在场):

1. 操作员打开 Grafana 大盘,显示 Worker Pod 池(假设 5 个副本)
2. 触发持续压测脚本(每秒 50 提交)
3. 在管理后台点击「随机杀一个 Worker」按钮
4. 观察:
   - Pod 池水位从 5 → 4(K8s 检测到 Pod 异常)
   - 该 Pod 上未完成的任务 30s 后被 dispatcher 重新派发
   - Sentinel 大盘显示派发节点暂时熔断,自动恢复
   - K8s HPA / Deployment 控制器 30s 内拉起新 Pod,水位回到 5
   - 全程提交服务可用,前端用户感知到的只是少数提交延迟略增
